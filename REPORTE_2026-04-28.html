De prototipo frágil a MVP demo-ready

Resumen para Matías: qué tocamos hoy en backend, n8n, frontend y Postgres, qué estaba roto antes, y cómo quedó después.

MVP funcionando end-to-end · 100% local

TL;DR — qué cambió

  • El orquestador de N8N pasó de tener URLs hardcodeadas y una API key de Gemini en texto plano a usar un nodo Config y credenciales nativas.
  • Agregamos un endpoint nuevo en FastAPI (/api/v1/rag-stores) sin el cual el agente no podía responder "qué archivos tenés".
  • Creamos un proxy en el backend (/api/v1/agent/chat) para que el frontend no hable directo con N8N — separación limpia para producción.
  • El agente ahora tiene memoria conversacional persistida en Postgres, sobrevive a reinicios de N8N.
  • Toda conversación se guarda en chat_history: si recargás el browser, el chat sigue ahí.
  • Frontend nuevo: vista Chat de 3 columnas (auditorías / chat / evidencia) con animación de pasos del agente.

● BACKEND · FastAPI + SQLAlchemy

1Backend: dos endpoints nuevos + persistencia

El backend tenía un agujero importante: el orquestador de N8N quería hablar con un endpoint /rag-stores/ que no existía. Además, el frontend tenía que hablar directo con N8N, lo cual no escala a producción.

1.1 — Endpoint GET /api/v1/rag-stores/

Antes

El agente no podía listar stores

La tool listar_stores del orquestador hacía GET http://localhost:8000/api/v1/rag-stores/, pero esa ruta nunca existió en FastAPI. Devolvía 404 silencioso.

Faltaba el router; existían el modelo y el repository pero nadie los exponía.
Después

Endpoint creado y validado

Nuevo router en app/api/v1/endpoints/rag_stores.py, service en app/services/rag_store_service.py, método list_all() en el repo. curl devuelve 200 con los 2 stores que ya estaban en la BD.

Respeta la arquitectura en capas: endpoint → service → repository.

1.2 — Endpoint POST /api/v1/agent/chat (proxy + persistencia)

Antes

Frontend hablaba directo con N8N

El plan era pegarle desde el browser a http://localhost:5678/webhook/router. Funciona local pero rompe producción: el frontend tendría que conocer la URL interna de N8N, sin auth, sin proxy.

No se separa la capa de presentación de la de orquestación. Mal para deploy.
Después

Backend proxea + guarda en BD

Nuevo endpoint en app/api/v1/endpoints/agent.py: recibe del frontend, valida la sesión, persiste la pregunta en chat_history, llama al orquestador, persiste la respuesta, devuelve. El frontend solo conoce localhost:8000.

Para producción solo cambiamos n8n_base_url en el .env del backend.

1.3 — Endpoint GET /api/v1/sessions/{id}/messages

Permite al frontend cargar el historial al abrir una sesión. Refresh-safe: recargás la página y todo el chat sigue ahí.

Snippet del nuevo endpoint /agent/chat

# app/api/v1/endpoints/agent.py
@router.post("/chat", response_model=AgentChatResponse)
async def agent_chat(body: AgentChatRequest, db: DBSession = Depends(get_db)):
    # 1. validar que la sesión existe (FK chat_history → sessions)
    if not session_repo.get(body.session_id):
        raise HTTPException(404, f"Sesion {body.session_id} no existe")

    # 2. persistir pregunta antes de llamar al agente
    history = chat_repo.create(session_id=body.session_id, question=body.mensaje, ...)

    # 3. proxy al orquestador de N8N (httpx async, no bloquea)
    result = await n8n_client.trigger_agent(mensaje=body.mensaje, session_id=body.session_id)

    # 4. persistir respuesta y devolver
    chat_repo.update_answer(history, answer=result["respuesta"], sources=[])
    return AgentChatResponse(...)
● N8N · Orquestador + WF0 + WF1

2N8N: configuración portable + sin secretos en el JSON

El orquestador y los flujos de upload tenían dos problemas serios: URLs hardcodeadas (rompen en producción) y la API key de Gemini en texto plano dentro del JSON exportado, que vive en git.

2.1 — Nodo Config al inicio del orquestador

Antes

URLs localhost hardcodeadas en cada tool

Las 3 tools del agente tenían pegado http://localhost:5678/webhook/... y http://localhost:8000/api/v1/... en el código JS de cada nodo.

Para deploy había que editar 8 lugares distintos. Imposible de mantener.
Después

Un solo nodo Config con todas las constantes

Un nodo Code al inicio expone api_base, n8n_base, gemini_model, temperature, memory_limit. Las tools leen con $('Config').first().json.api_base. Migrar a producción = editar un solo nodo.

No usamos $vars porque solo está en el plan pago de N8N. No usamos $env porque ata el flujo al entorno donde corre N8N. Config funciona idéntico en free y pago, local y prod.

2.2 — API key de Gemini fuera del JSON

Antes — bug crítico

Key en texto plano dentro del JSON

WF0 y WF1 tenían ?key=AIzaSyB1-nUs2j... directo en la URL del HTTP Request. Como los JSON están en git, la key estaba expuesta.

Cualquiera con acceso al repo veía la key. Riesgo de fuga + costos.
Después

Credencial nativa de N8N + key rotada

1) Rotaste la key vieja en Google Cloud. 2) Los HTTP Request ahora usan "Predefined Credential Type → Google PaLM API" con la credencial existente Google Gemini(PaLM) Api account. La key vive solo en la credencial encriptada de N8N. grep AIzaSy no encuentra nada en el repo.

Cuando exportes JSON solo viaja el ID/nombre de la credencial. Al importar en prod, N8N pide vincular con la equivalente.

2.3 — Memoria del agente conectada a Postgres

Sin nodo de memoria → agente sin contexto

Le decías "mi nombre es Harry" y al turno siguiente respondía "no tengo memoria de conversaciones previas".

Los AI Agents de N8N requieren un nodo conectado al puerto ai_memory. Estaba vacío.

Memory Buffer Window (en RAM del proceso N8N)

Funciona, pero al reiniciar N8N se pierde toda la memoria de todas las sesiones.

Bueno para probar el patrón. Insuficiente para una demo seria.

Memory Postgres Chat — sobrevive restarts

Nodo memoryPostgresChat con sessionKey = $('Extraer datos').first().json.session_id y ventana de 10 turnos. Crea la tabla n8n_chat_histories automáticamente, usa la credencial Postgres existente.

Verificado: reiniciamos N8N, preguntamos "cómo me llamo", respondió "Harry" — la memoria sobrevivió.

● POSTGRES · estado de las tablas

4Postgres: dos tablas para el chat (intencional)

Hoy hay dos tablas que guardan conversaciones, cada una con un dueño distinto. Es así por ahora porque cada parte se maneja con una tecnología diferente.

TablaDueñoPara qué
chat_history FastAPI El historial visible para el usuario. Lo escribe /agent/chat, lo lee /sessions/{id}/messages. Es lo que ve el frontend.
n8n_chat_histories N8N (auto-creada) La memoria del agente que el AI Agent inyecta en el prompt. Schema lo define n8n.

En la solución final esto se unifica: una sola tabla messages en nuestro schema, con auditoria_id, user_id, sources, etc. El backend pasaría las últimas N filas como contexto al agente y N8N quedaría stateless. Lo dejamos para fase 2 cuando metamos auth y multi-tenancy.

6Pendientes para fase 2

A

Modelo de dominio bancario

Introducir entidades Auditoria, User, AuditoriaMembership. Hoy la "sesión" es un identificador genérico — en producción cada auditoría dura semanas, tiene varios auditores asociados, alcance, fechas, estado.

B

Autenticación

Probablemente OAuth/OIDC contra el AD del banco (Azure AD / Keycloak). No agregamos JWT propio porque el banco va a exigir lo suyo.

C

Migraciones Alembic

Hoy el schema se crea con create_tables() al arrancar. En cuanto haya datos reales, se rompe.

D

Solución "final" del estado conversacional

Unificar chat_history y n8n_chat_histories en una sola tabla messages en nuestro schema. Backend dueño del estado, N8N stateless.

E

Streaming real del agente

Los pasos del chat hoy son heurísticos. Para mostrar lo que el agente realmente hace en tiempo real hay que stremear desde N8N execution log via SSE/WebSocket.
